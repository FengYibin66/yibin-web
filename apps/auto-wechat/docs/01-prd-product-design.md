## 1. 项目概述

### 1.1 一句话定位

**AI 驱动的微信公众号自动化运营系统**：采集全球 AI 资讯 → 智能筛选与内容生产 → 排版 → 发布与数据监控闭环。

**MVP 策略**：先跑通「一键触发」的一次性全链路（抓取 → 生产 → 发布），验证质量与流程；定时调度、每日自动推送后置到 V1。

### 1.2 背景与痛点

| 痛点 | 现状 | 目标 |
|------|------|------|
| 资讯追踪 | 来源分散（外网 + 中文），人工刷 RSS/推特耗时长 | 一键触发采集，精选 Top 10 |
| 内容生产 | 从选题到排版质量不稳定，产能有限 | 多 Agent 协作：写作 + **排版专家**，输出可直接发布的成稿 |
| 数据运营 | 后台数据分散，缺乏趋势分析与行动建议 | 统一看板 + 每日/每周 AI 分析报告 |

### 1.3 成功指标（North Star）

| 指标 | MVP 目标 | V1 目标 |
|------|----------|---------|
| **一次性链路成功率** | 手动触发后端到端成功 ≥ 90% | ≥ 99% |
| **全链路耗时** | 抓取 → 发布 ≤ 15 分钟 | ≤ 10 分钟 |
| 有效资讯数 | Top 10，重复率 < 10% | Top 10，重复率 < 5% |
| 内容可用率 | 排版后少改即用 ≥ 70% | ≥ 85% |
| 定时推送准时率 | —（MVP 不做） | ≥ 99% |
| 阅读/分享 | 建立基线后环比 +10% | 环比 +20% |

## 3. MVP 核心：一次性全链路

### 3.0 链路定义

MVP 的首要交付物是一条 **可手动触发、端到端跑通** 的流水线，而非定时任务。

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    一次性全链路（Run Once Pipeline）                      │
│                         触发方式：API / CLI / Web 按钮                   │
└─────────────────────────────────────────────────────────────────────────┘

  ① 采集          ② 筛选           ③ 内容生产              ④ 发布
  ─────          ─────           ──────────              ─────
  多源抓取   →   Top 10     →    Editor 选题        →   微信草稿箱
  清洗去重       摘要+配图         Writer 写作              或
                                  Layout 排版专家          直接发布
                                  Reviewer 质检
                                       ↓
                                  人工确认（可选）
```

**与定时方案的关系：**

| 能力 | MVP | V1 |
|------|-----|-----|
| 手动触发全链路 | ✅ P0 | ✅ |
| 分步手动触发（仅采集 / 仅生产） | ✅ P1 | ✅ |
| Cron 定时采集 | ❌ | ✅ |
| 定时推送 Top 10 | ❌ | ✅ |
| 定时自动生成 + 发布 | ❌ | ✅（可配置） |

**触发入口（MVP 至少一种）：**

- `POST /api/v1/pipeline/run` — 跑完整链路
- CLI：`python -m app.cli run-pipeline`
- Web UI：「一键生成并发布」按钮

**链路输出：**

- 本次 Run 的 `run_id`、各阶段状态、耗时
- Top 10 Digest 快照
- 成稿预览（HTML）
- 发布结果（草稿箱 media_id / 发布链接）

## 4. 功能模块

### 4.1 模块 A：AI 资讯采集

#### 4.1.1 功能描述

从全网（外网 + 中文源）采集 AI 相关资讯，经过去重、排序、 enrichment 后输出 Top 10，作为内容生产输入。**MVP 阶段由全链路触发，不做独立定时任务。**

#### 4.1.2 功能清单

| ID | 功能 | 优先级 | 说明 |
|----|------|--------|------|
| A-01 | 多源采集 | P0 | RSS / API / 搜索 API / 定向爬虫 |
| A-02 | 清洗标准化 | P0 | 统一字段：标题、URL、来源、时间、摘要、正文、图片 |
| A-03 | 去重聚类 | P0 | URL 去重 + 标题相似度 + 事件聚类 |
| A-04 | 重要性排序 | P0 | LLM + 规则混合，输出 Top 10 + 排序理由 |
| A-05 | 摘要与翻译 | P0 | 英文源自动中文摘要（100–150 字） |
| A-06 | 配图 | P0 | og:image / 正文首图；无图时默认封面 |
| A-07 | 链路内嵌执行 | P0 | 作为 Pipeline 第一步，随 `run` 触发 |
| A-08 | 独立采集触发 | P1 | 仅跑采集 + Top 10，不生产不发布 |
| A-09 | 人工 override | P1 | Web UI 替换/删除/手动添加条目 |
| A-10 | 历史归档 | P1 | 按 Run / 日期浏览 Digest |
| A-11 | 定时采集 | V1 | Cron 每日固定时间 |
| A-12 | 定时推送 Top 10 | V1 | 企微 / 飞书 / 邮件 |
| A-13 | 源扩展 | P2 | 可视化配置新数据源 |

#### 4.1.3 数据源规划

**MVP（Phase 1，5–8 源）：**

| 源 | 类型 | 语言 | 备注 |
|----|------|------|------|
| Hacker News | RSS/API | EN | AI 相关 filter |
| ArXiv cs.AI | RSS | EN | 论文速递 |
| Reddit r/MachineLearning | RSS | EN | 社区热点 |
| 机器之心 | RSS/爬虫 | CN | 中文 AI 媒体 |
| 量子位 | RSS/爬虫 | CN | 中文 AI 媒体 |
| OpenAI Blog | RSS | EN | 官方动态 |
| Google News (AI 关键词) | RSS | EN/CN | 广覆盖补充 |

**V1 扩展（15+ 源）：**

TechCrunch AI、The Verge AI、MIT TR、Anthropic/Google/Meta 官方博客、36氪 AI、InfoQ、GitHub Trending AI、Product Hunt AI、Twitter/X 列表等。

#### 4.1.4 Digest 输出形态（链路内展示，V1 才独立推送）

```
📰 AI 要闻 Top 10 | Run #abc123 | 2026-05-31 14:32

━━━━━━━━━━━━━━━━━━━━
1. [标题]
   📝 一句话摘要（100 字内）
   💡 为什么重要：…
   🔗 来源名 | 原文链接
   🖼️ [配图缩略图]
…
10. …

━━━━━━━━━━━━━━━━━━━━
📊 本次采集：128 条 → 去重 45 条 → 精选 10 条
⏱️ 耗时：采集 2m12s | 排序 18s
```

#### 4.1.5 验收标准

- [ ] 作为 Pipeline 第一步，随 `run` 触发可独立完成采集 + Top 10
- [ ] Top 10 无明显重复（同一事件 ≤ 1 条）
- [ ] 每条含：标题、中文摘要、来源、链接、配图（或默认图）
- [ ] 英文源摘要为流畅中文
- [ ] 部分源失败不阻塞链路（降级：用已采集数据出 Top N，N ≥ 5）

### 4.2 模块 B：微信公众号内容生产

#### 4.2.1 功能描述

基于 Top 10 资讯，由多 Agent 协作生成符合微信公众号规范的成稿：**Writer 负责内容与结构，排版专家负责视觉呈现**，经质检后可发布。**MVP 作为全链路环节串联执行。**

#### 4.2.2 Agent 分工

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| **Editor** | 选题 + 大纲 | Top 10 Digest | 选题、文章类型、结构化大纲 |
| **Writer** | 内容写作 | 大纲 + 素材 | 标题候选、导语、正文（Markdown/纯文本）、引用来源 |
| **排版专家（Layout）** | 微信视觉排版 | Writer 输出 + 图片素材 | 微信兼容 HTML、封面、插图位置、样式规范 |
| **Reviewer** | 内容质检 | 排版后成稿 | 通过 / 修改建议（事实、来源、敏感词） |
| **Publisher** | 上传发布 | 终稿 | 微信草稿箱 media_id / 发布结果 |

**Writer 与排版专家的边界：**

| 维度 | Writer | 排版专家 |
|------|--------|----------|
| 关注点 | 写什么、怎么写、逻辑结构 | 怎么呈现、好不好读、是否符合微信规范 |
| 输出 | Markdown / 结构化 JSON | 微信 HTML + 样式内联 |
| 不负责 | 字体字号、段落间距、移动端适配 | 观点、事实、行文逻辑 |

#### 4.2.3 内容形态

| 形态 | 频率建议 | 字数 | 说明 |
|------|----------|------|------|
| **AI 日报合集** | 日更/隔日 | 800–1200 | Top 5–10 精简版，适合快讯 |
| **单篇深度解读** | 2–3 次/周 | 1500–3000 | 选 1 条重磅新闻深度写 |
| **工具/产品测评** | 1 次/周 | 1000–2000 | 新 AI 工具上手体验 |
| **论文速读** | 1 次/周 | 800–1500 | ArXiv 热门论文通俗解读 |

#### 4.2.4 功能清单

| ID | 功能 | 优先级 | 说明 |
|----|------|--------|------|
| B-01 | 选题 Agent（Editor） | P0 | 从 Top 10 确定选题与文章类型 |
| B-02 | 大纲生成 | P0 | 结构化大纲（含小标题） |
| B-03 | 正文写作（Writer） | P0 | 符合品牌调性，输出 Markdown |
| B-04 | 标题优化 | P0 | 3–5 个候选标题 + 推荐理由 |
| B-05 | 摘要/导语 | P0 | 100 字内导读 |
| B-06 | **排版专家（Layout）** | **P0** | 见 4.2.5 |
| B-07 | 封面图 | P0 | 900×500 或 2.35:1，排版专家统一规范 |
| B-08 | 正文插图 | P0 | 位置、尺寸、对齐由排版专家决定 |
| B-09 | AI 质检（Reviewer） | P0 | 事实、来源、敏感词；排版规范检查 |
| B-10 | 发布（Publisher） | P0 | 上传素材 + 草稿箱 / 直接发布 |
| B-11 | 链路内预览 | P0 | Run 结束前展示 HTML 预览 |
| B-12 | 人工确认闸门 | P1 | 发布前暂停等人点「确认发布」 |
| B-13 | 独立生产触发 | P1 | 指定 Digest 仅跑生产环节 |
| B-14 | 内容日历 | V2 | 周/月选题规划 |

#### 4.2.5 排版专家（Layout Agent）规范

排版专家是独立 Agent，接收 Writer 的结构化内容，输出**可直接进微信编辑器或 API 发布**的成稿。

**排版职责：**

| 类别 | 具体要求 |
|------|----------|
| **段落** | 段间距统一；单段 ≤ 4 行（移动端）；避免 wall of text |
| **标题** | 层级清晰（H2/H3）；小标题加粗或品牌色区分 |
| **引用** | 金句 / 来源引用使用统一引用块样式 |
| **列表** | 有序/无序列表格式一致，序号与缩进规范 |
| **代码** | 代码块等宽字体、背景色、横向滚动（如有技术内容） |
| **图片** | 宽度 100% 或 90% 居中；图注小字灰色；与上下文间距 |
| **链接** | 来源链接可点击；外链样式统一 |
| **分隔** | 章节间使用分隔线或留白，节奏感清晰 |
| **封面** | 900×500px，标题可读，品牌色 / 字体一致 |
| **尾部** | 固定「来源声明」「推荐阅读」等栏目样式 |

**排版专家输入/输出：**

```json
// 输入（来自 Writer）
{
  "title": "主标题",
  "title_candidates": ["...", "..."],
  "summary": "导语",
  "body_markdown": "...",
  "sources": [{"name": "...", "url": "..."}],
  "images": [{"url": "...", "caption": "...", "suggested_position": "after_section_2"}]
}

// 输出
{
  "title": "最终标题",
  "cover_image_url": "...",
  "body_html": "<section>...</section>",
  "layout_notes": "排版说明（如有特殊处理）"
}
```

**品牌排版模板 `[TBD]`：**

- 主色 / 辅色
- 正文字号（建议 15–16px）、行高（1.75–1.8）
- 引用块、代码块、分隔线 CSS 片段
- 1–2 篇标杆排版范文作为 Few-shot

#### 4.2.6 内容质量标准

| 维度 | 标准 |
|------|------|
| **标题** | ≤ 20 字，有信息增量，避免标题党 |
| **导语** | 100 字内点明「为什么值得读」 |
| **正文** | 逻辑清晰：背景 → 要点 → 影响 → 展望 |
| **来源** | 必须标注原文链接与来源名 |
| **时效** | 标注事件发生/发布时间 |
| **排版** | 移动端阅读友好；样式统一；图片不溢出；来源区格式固定 |

| **禁止** | 未核实谣言、大段抄袭、过度营销、敏感违规内容 |

#### 4.2.7 品牌调性 Prompt 要素 `[TBD]`

- 人称：你 / 我们 / 客观第三人称
- 风格：专业但不晦涩 / 轻松幽默 / 严肃深度
- 固定栏目：如「今日一词」「工具推荐」
- 示例文章：提供 2–3 篇范文作为 Writer Few-shot
- **排版范文**：提供 1–2 篇排版效果标杆作为 Layout Few-shot

#### 4.2.8 发布流程（一次性全链路）

```
触发 Pipeline Run
    ↓
① Collector → Ranker → Enricher     输出 Top 10 Digest
    ↓
② Editor Agent                      选题 + 大纲
    ↓
③ Writer Agent                      标题 + 导语 + 正文（Markdown）
    ↓
④ 排版专家 Layout Agent             微信 HTML + 封面 + 插图排版
    ↓
⑤ Reviewer Agent                    内容与排版质检 → 通过 / 驳回
    ↓
⑥ [可选] 人工确认闸门              预览 → 确认 / 取消
    ↓
⑦ Publisher                         微信草稿箱 / 直接发布
    ↓
输出 run 结果（预览链接、media_id、耗时）
```

**驳回回退规则：**

- Reviewer 驳回「内容问题」→ 回到 Writer（带上修改意见）
- Reviewer 驳回「排版问题」→ 回到排版专家（Writer 输出不变）
- 最多循环 2 次，仍失败则 Run 标记 `failed` 并保留中间产物供人工处理

#### 4.2.9 验收标准

- [ ] 一次 `run` 可从抓取走到发布（或草稿箱），全链路 ≤ 15 分钟
- [ ] Writer 输出与 Layout 输出职责分离，Layout 不改动核心观点
- [ ] 排版后 HTML 在微信编辑器 / 手机预览无错乱
- [ ] 标题、导语、正文、来源链接、封面齐全
- [ ] 质检能区分内容问题 vs 排版问题并回退到对应 Agent
- [ ] 发布成功后返回可追踪的 `media_id` 或文章链接

### 4.3 模块 C：数据监控看板与自动分析

#### 4.3.1 功能描述

采集公众号后台数据，在看板展示核心指标，并由 AI 生成分析报告。**MVP 不做，V2 实现；MVP 仅记录 Pipeline Run 与发布结果。**

#### 4.3.2 功能清单

| ID | 功能 | 优先级 | 说明 |
|----|------|--------|------|
| C-00 | Pipeline Run 历史 | P0（MVP） | 每次 run 的状态、耗时、各 Agent 输出 |
| C-01 | 指标采集 | V2 | 阅读、分享、收藏、在看、涨粉、取关 |
| C-02 | 文章维度 | V2 | 单篇 PV/UV、完读率、分享率 |
| C-03 | 总览看板 | V2 | 今日/7 天/30 天核心 KPI |

## 5. 非功能需求

### 5.1 可靠性

- Pipeline 单阶段失败自动重试（最多 3 次，指数退避）
- 采集部分失败不阻塞全链路（降级：用已采集数据出 Top N，N ≥ 5）
- Run 失败保留中间产物，支持从失败阶段重跑（V1）
- 关键错误即时通知（企微/飞书）

### 5.2 合规与安全

- 爬虫遵守 robots.txt，控制频率（≤ 1 req/s/域）
- 转载注明来源与原文链接
- 敏感词过滤（政治、色情、谣言）
- API Key 环境变量存储，不入库明文
- 微信 Access Token 自动刷新

## 6. MVP 范围边界

### 6.1 MVP 包含（一次性全链路）

- **`POST /api/v1/pipeline/runs`**（Web 按钮 / API）一键触发
- 5–8 个 RSS/API 数据源采集 → 去重 → LLM Top 10
- 内容生产 Agent 链：**Editor → Writer → 排版专家 → Reviewer**
- 1 种内容模板（AI 日报合集）
- 排版专家输出微信兼容 HTML + 封面
- **半自动发布**：Publisher 调用 **微信草稿箱 API**（`draft/add`，已验证）→ 运营者在 **mp.weixin.qq.com 草稿箱点「发表」**
- 成稿预览（Web 控制台，sandbox iframe，非 v-html）
- Pipeline Run 历史页：状态、各阶段输出、`media_id`、草稿箱指引
- 个人未认证号 **不可用** `freepublish` 直发（errcode 48001），MVP 不依赖直发

### 6.2 MVP 不包含

- Cron 定时采集 / 定时推送
- Top 10 独立推送到企微/邮件（V1）
- 完整数据看板与 AI 分析报告
- 内容日历、竞品监控
- 多内容形态模板（深度文、工具测评等）

### 6.3 后续版本

| 版本 | 新增能力 |
|------|----------|
| V1 | 定时采集 + Top 10 推送；15+ 源；从失败阶段重跑；排版模板库 |
| V2 | 数据看板 + 指标采集 + AI 分析 + 告警 |
| V3 | 多内容形态；内容日历；A/B 标题；竞品监控 |
