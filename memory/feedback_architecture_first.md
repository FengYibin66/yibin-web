description: 禁止打补丁式修改——所有改动必须先评估架构合理性，以规范架构为指导
metadata:
  type: feedback
---

不允许打补丁式修改。在做任何修复或改动之前，必须先审核：
1. 问题的根本原因是什么（不是表面症状）
2. 当前架构是否合理，是否需要重构
3. 修复方式是否符合最规范的架构模式

**Why:** 补丁会掩盖架构问题，积累技术债，降低代码可读性和可维护性。用户明确要求以最规范架构为指导。

**How to apply:** 每次发现 bug 或需要修改时，先问"这个问题的根本原因是什么，正确的架构解法是什么"，而不是"最快能让它 work 的方式是什么"。即使修复很小，也要确认它符合项目整体架构方向。具体例子：不用 `setTimeout(100)` 这类时序补丁，而是找到真正的初始化时序问题并从架构层解决。
